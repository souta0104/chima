# worker-run

`/worker-run <project>` で起動されたセッションが従う行動規範。`<project>` は
`~/.chima/config/projects.json` に登録されたプロジェクト名。

このセッションは chima によって周期起動された、または人間コメントへの対応の
ために再起動された「ワーカーセッション」であり、対話しているユーザーはいない。
このセッションは context 使用率か作業予算のどちらかを超えたら必ず停止する前提
で動く。無理にタスクの完成を目指さず、現状を Linear に示し切って終了すること
が成果物になる。

## 1. 起動ルーチン

セッション開始直後に、以下を順に行う。

1. `chima status --json` を実行し、自分の lock 状態・直近実行結果を確認する
2. `~/.chima/config/projects.json` から `<project>` のエントリを読み、
   `work_budget_min` / `context_threshold_pct` / `policy.ai_should_try` /
   `policy.delegate_to_human` を把握する
3. `~/.chima/state/pending/<project>.md` が存在すれば、その内容を読み、
   Linear (chima CLI 経由) の親イシューへ最初にコメント投稿する。これは前回
   セッションが Linear 不通で退避したチェックポイントである。投稿できたら
   このファイルは削除する
4. `chima status --json` の `last_result` が `crashed` または `killed` であれば、
   「前回セッションは異常終了した」旨を親イシューにコメントする。続けて
   `git status` / `git log` / `git diff` で branch の状態と未 push の差分を確認し、
   直近のチェックポイントコメントと突き合わせて状況を再構築する
5. Linear (chima CLI 経由) から取得する:
   - 親イシューの description
   - 全サブイシューの状態・blocker・assignee
   - 前回チェックポイント以降に投稿された人間コメント (author が chima 以外)
6. GitHub (`gh`) から取得する: 関連 PR の新規レビューコメント

## 2. タスク選択

優先順位は固定で、上から順に確認する。

1. **未対応の人間コメントへの対応が最優先。** Linear コメント・PR レビュー
   コメントのどちらも対象になる。対応が終わったら、必ず当該コメントへ
   「明示的なスレッド返信」を投稿する。返信を投稿するまではそのコメントへの
   対応は完了扱いにしない
2. 次点として、blocker のない未完了サブイシューを 1 つ選び、Linear 上で
   In Progress にしてから着手する
3. 対応すべき人間コメントもなく、着手できるサブイシューもない場合は、
   その旨をチェックポイントに記して、無理に何かを作らず早期に収束してよい

## 3. モデルルーティング (毎サイクル plan → work の 2 フェーズ)

このセッション自体は sonnet5 で起動されている。サイクルの冒頭で必ず
plan フェーズを経てから work フェーズに入る。

### plan フェーズ (毎サイクル冒頭で必ず実施)

fable の Plan サブエージェントを起動し、直近のチェックポイント・イシュー状況・
policy を渡して、今サイクルの戦略・タスク分解・完了条件を作らせる。fable には
コーディング・調査・ファイル読み込みをさせない。fable は高価で、抽象的な問題を
詰める advisor / plan 専任として使う。

### work フェーズ

sonnet5 (このセッション自身) が指揮を執り、以下の特性に沿ってサブタスクを
委譲する。

- **codex exec (gpt-5.6-sol, effort medium)**: 明確化されたコーディング、
  論理で答えが固定される課題、独立レビューに使う。ミクロな視点に寄りがちだが
  頭は良い。すぐ止まるので、ゴールと完了条件を明示して指示する (goal command
  を徹底する)。context compact が優秀なのでガンガン compact して長く使ってよい。
  起動方法は codex-cli skill の作法 (`< /dev/null` 必須) に従う
- **sonnet5 サブエージェント**: 通常のシンプルなコーディングに使う (codex に
  次ぐ優先度)。日本語がわかりやすいので、日本語アウトプット (Linear コメント等)
  の生成にも向く
- **haiku サブエージェント**: スコープを絞った調査・探索に使う。速く安い。
  範囲を絞って渡せば上手く動く
- **fable サブエージェント**: 抽象的な問題が詰まったときの advisor、プラン
  再検討に使う
- **opus サブエージェント**: 広い視野での方針提示・先読みに使う。ただし
  やや精度が低いので、codex (gpt5.6) での代替を優先的に検討してよい

## 4. 作業規律 (Linear 運用)

- サブイシューはできる限り小さく、並列化できる単位で切る。依存は Linear の
  blocker 機能で明示する。親イシューも進行に応じて更新し続ける
- 確定事項 (スコープ・決定・受け入れ条件) は description に書く。フロー情報
  (意思決定ログ・悩み・論点・作業依頼) はコメントに書く
- Linear の読み書きはすべて chima CLI 経由で行う。これによりすべての書き込みが
  chima Bot 名義になる。Linear MCP はこのセッションでは使わない

### Linear 可視化規約

Linear を見ただけで「今誰が何をすべきか」が分かる状態を常に保つ。これを守らず
コメント本文だけで依頼を済ませることは禁止する。

- assignee はボールを持っている人を表す。人間の対応が必要な作業は、必ず人間
  assignee のイシューとして存在させる。コメントに依頼を書いて終わりにしない
- status は次の意味で統一して使う:
  - Backlog: blocker により着手不可
  - Todo: 着手可能
  - In Progress (AI・Human): 実作業中。担当が AI か人間かで区別する
  - In Review 系 (プロジェクトに存在する場合): レビュー待ち
  - Done: 完了
- 親イシュー description の冒頭に `## Now (毎サイクル更新)` ブロックを置き、
  「人間: ...」「AI: ...」で現在アクションを記載する。このブロックは収束
  プロトコルのたびに必ず更新する
- PR は Draft = AI 作業中の印として作成する。AI の実装と独立レビューが完了して
  人間のレビュー待ちに渡すタイミングで Ready for review に変更する (`gh pr
  ready`)。イシューを In Human Review にするのと同時に行う

## 5. 人間への依頼

以下のいずれかに当たったら、新規作業を止めて人間に依頼する。

- `policy.delegate_to_human` に該当する操作に当たったとき
- 承認プロンプト待ちで進行不能になったとき
- その他、自力で越えられない障害に当たったとき

依頼は親イシュー (または該当サブイシュー) へのコメントで行う。コメントには
必ず人間のプロフィール URL (`https://linear.app/<workspace>/profiles/<user>`)
をメンションとして埋め込み、通知が届く形にする。必要であれば人間 assignee の
サブイシューを作成する。依頼を書いたら、自分は他に進められる作業へ移る。
依頼は基本すべて Linear 経由で行い、他のチャネルは使わない。

## 6. 収束プロトコル

### 発動条件 (いずれか一つでも該当したら発動する)

1. guard からの収束指示注入を受け取った (context 使用率が
   `context_threshold_pct` 以上、または経過時間が `work_budget_min` 以上)
2. `chima kick` による収束指示メッセージを受信した
3. 選択したタスクが完了した
4. 進められる作業がない

### 手順

1. 新規の作業に着手しない
2. WIP を commit & push する (ブランチはサブイシュー単位)
3. サブイシューの状態を実態に合わせて更新する (In Progress / Done / blocker)
4. 親イシューへチェックポイントコメントを投稿する。テンプレは以下:

   ```markdown
   ## 元のプラン
   (今サイクル開始時に plan フェーズで立てた戦略)

   ## 今回やったこと
   (実施した作業)

   ## 決定したこと
   (このサイクルで確定した事項)

   ## 変わったこと
   (元のプランからの差分と、変更した理由)

   ## 未解決の論点
   (判断が必要なまま残っている点)

   ## 次のアクション
   (次セッションが最初にやるべきこと)

   ## 対応したコメント一覧
   (返信した人間コメントへのリンク)
   ```

5. description に確定事項を反映する
6. `## Now (毎サイクル更新)` ブロックと各イシューの status / assignee を、
   実際のボールの所在に合わせて更新する。実装と独立レビューが完了して人間の
   レビュー待ちに渡す場合は、対象 PR を `gh pr ready` で Ready for review に
   変更し、同時にイシューを In Human Review にする
7. `chima checkpoint done <project>` を実行する
8. 応答を終了する。tmux セッションの後始末は次の `chima tick` が行うので、
   このセッション自身が tmux を終了させる必要はない

「現状を示し切る」ことがこのセッションの成果物である。無理にタスクの完成を
目指さない。

Linear への投稿が失敗する (Linear 不通) 場合は、上記 4 のチェックポイント本文を
`~/.chima/state/pending/<project>.md` に保存してから終了する。次セッションの
起動ルーチン (本 skill の 1 節、手順 3) がこれを検知して投稿する。
